# 公司 / 项目管理模块规格

> 适配后端 spec：[../../backend/spec/task_management.md](../../backend/spec/task_management.md)（§公司 / §项目 两个小节）
> 入口：从任务列表操作条点 `管理公司项目` 按钮，弹出 `<el-dialog>`；
> 不走独立路由，不进侧边栏导航（与「管理类型」同级，均为宿主页的辅助维护入口）。
> 全局规则遵循 [./README.md](./README.md)，本文档只描述本模块特有的页面拆解、功能点交互与测试案例。

---

## 1. 整体页面结构拆解

### 1.1 入口与宿主

本模块不注册新路由，仅提供一个对话框组件 `components/CompanyProjectManager.vue`，由任务列表 `views/TaskList.vue` 操作条中的 `管理公司项目` 按钮触发（与 `管理类型` 按钮同行）。

> 选择此方案的理由：
> - 与「管理类型」保持一致（一个按钮 + 一个 dialog），降低用户认知负担；
> - 公司 / 项目是任务的依赖数据，平时只读消费，不需要独占页面；
> - 单用户本地版，写操作频次低，弹窗足够。
>
> 若 v0.2 出现「批量导入」「公司分组」等更重的需求，再考虑改成独立路由页。

### 1.2 对话框 DOM 结构

```
<el-dialog v-model="visible" title="公司 / 项目管理" width="800px">
  <el-tabs v-model="activeTab">
    ├─ Tab 「公司」
    │   ├─ 顶部新建行：[name 输入框] [notes 输入框] [+ 新建]
    │   └─ 公司列表 <el-table>
    │         列：name / notes / 创建日 / 操作（编辑 / 删除）
    │
    └─ Tab 「项目」
        ├─ 公司筛选：[公司下拉 clearable]
        ├─ 顶部新建行：[name 输入框] [notes 输入框] [+ 新建]
        └─ 项目列表 <el-table>
              列：name / 所属公司 / notes / 创建日 / 操作（编辑 / 删除）
</el-dialog>
```

### 1.3 模块涉及组件与 store

| 类型 | 名称 | 职责 |
|------|------|------|
| Component | `components/CompanyProjectManager.vue` | 公司 / 项目管理对话框（两 tab）；新增、编辑、删除 |
| Store | `stores/useCompanyStore.js` | 公司列表 CRUD 状态（已有，扩展 actions） |
| Store | `stores/useProjectStore.js` | 项目列表 + 按公司分组 CRUD 状态（已有，扩展 actions） |
| API | `api/companies.js` / `api/projects.js`（已有） | 后端接口封装 |

> `useCompanyStore.companies` / `useProjectStore.byCompany[company_id]` 同时是任务表单下拉的字典来源；本模块任何写操作成功后必须**同步刷新对应字典字段**，保证下拉立即可用。

## 2. 页面的功能点

### 2.1 功能点：打开管理对话框

#### 入口
- 任务列表操作条 `管理公司项目` 按钮。

#### 静态展示规则
- 弹窗宽度 800px，居中；
- 标题「公司 / 项目管理」；
- 默认 Tab = `company`（公司）；
- 公司 Tab 顶部「+ 新建」行：name 输入（200px）+ notes 输入（300px）+ `+ 新建` 按钮；
- 公司列表列：`name / notes / 创建日 / 操作`；
- 项目 Tab 顶部公司筛选下拉（clearable，默认空 = 显示全部项目）+ 新建行：name 输入（200px）+ notes 输入（300px）+ `+ 新建` 按钮；
- 项目列表列：`name / 所属公司 / notes / 创建日 / 操作`；
- 列表为空显示「暂无数据」占位。

#### 交互逻辑：打开

| 阶段 | 内容 |
|------|------|
| 操作前 | 主页正常展示 |
| 触发动作 | 点击 `管理公司项目` |
| 接口请求 | 并行：`GET /api/companies?page=1&size=100`<br>`GET /api/projects?page=1&size=100` |
| 成功逻辑 | 弹窗打开，两 Tab 数据就绪 |
| 失败逻辑 | 弹窗不打开 + `ElMessage.error`（沿用全局 5xx 文案） |

> 公司 / 项目总量小（单用户本地），不分页，size=100 一次性拉全。如果未来单类超过 100 条再升级分页参数。

#### 交互逻辑：切换 Tab

| 阶段 | 内容 |
|------|------|
| 操作前 | 弹窗已打开 |
| 触发动作 | 点击 Tab 头 |
| 接口请求 | 无（数据已加载） |
| 成功逻辑 | Tab 内容切换，已填写但未提交的新建行数据保留 |
| 失败逻辑 | — |

#### 交互逻辑：项目 Tab 切换公司筛选

| 阶段 | 内容 |
|------|------|
| 操作前 | 项目列表已渲染 |
| 触发动作 | 顶部公司下拉切换 |
| 接口请求 | 无（数据已全量加载到 store，前端按 `company_id` 过滤即可） |
| 成功逻辑 | 项目列表按所选公司过滤；下拉清空时显示全部 |
| 失败逻辑 | — |

---

### 2.2 功能点：新建公司

#### 静态展示规则
- 字段：

| 字段 | 控件 | 必填 | 长度 | 默认值 |
|------|------|------|------|--------|
| name | `<el-input>` | ✓ | 1-128 | 空 |
| notes | `<el-input>` | — | ≤ 4000 | 空 |

- 底部按钮：「+ 新建」；
- 提交中按钮 loading 并禁用。

#### 交互逻辑

| 阶段 | 内容 |
|------|------|
| 操作前 | name / notes 已填 |
| 触发动作 | 点击「+ 新建」 |
| 接口请求 | `POST /api/companies` body（snake_case）：<br>`{ name, notes? }` |
| 成功逻辑 | 公司列表追加新行；输入框清空；`useCompanyStore.companies` 同步追加；`ElMessage.success('创建成功')` |
| 失败逻辑 | 422 → name 字段红字（`msg` 文案）；<br>409 → `name` 字段红字「该公司名称已存在」；<br>5xx → 顶部 toast |

---

### 2.3 功能点：编辑公司

#### 静态展示规则
- 点击行内 `编辑` 进入行内编辑态：name / notes 输入框 + `保存` / `取消` 按钮；
- 一次只允许一行处于编辑态；
- 保存中：行内按钮 loading。

#### 交互逻辑：进入编辑

| 阶段 | 内容 |
|------|------|
| 操作前 | 公司列表渲染中 |
| 触发动作 | 点击行内 `编辑` |
| 接口请求 | 无 |
| 成功逻辑 | 该行变为编辑态；其他行若有编辑态则先退出（数据丢弃，不弹 dirty 提示——列表行编辑频次低） |
| 失败逻辑 | — |

#### 交互逻辑：保存

| 阶段 | 内容 |
|------|------|
| 操作前 | 行内编辑态 |
| 触发动作 | 修改字段，点击 `保存` |
| 接口请求 | `PUT /api/companies/{id}` body：`{ name, notes? }` |
| 成功逻辑 | 列表行更新；`useCompanyStore.companies` 中对应记录替换；`ElMessage.success('更新成功')` |
| 失败逻辑 | 422 → 行内字段红字；409 → 行内 name 红字 + toast「该公司名称已存在」 |

#### 交互逻辑：取消

| 阶段 | 内容 |
|------|------|
| 操作前 | 行内编辑态 |
| 触发动作 | 点击 `取消` |
| 接口请求 | 无 |
| 成功逻辑 | 行退出编辑态，原值恢复 |
| 失败逻辑 | — |

> 列表行内编辑不弹 dirty 二次确认（与任务类型管理一致）；若用户改了一半点取消，原值直接恢复。

---

### 2.4 功能点：删除公司

#### 静态展示规则
- 行内 `删除` 按钮：hover 显示危险色；
- 点击后立即触发 `ElMessageBox.confirm` 二次确认；
- 文案：`确定删除「{name}」？若已被项目或任务引用将无法删除。`

#### 交互逻辑：用户确认

| 阶段 | 内容 |
|------|------|
| 操作前 | 确认弹窗已打开 |
| 触发动作 | 点击「确定」 |
| 接口请求 | `DELETE /api/companies/{id}` |
| 成功逻辑 | 列表行移除；`useCompanyStore.companies` 同步移除；其下所有项目**不级联删除**，但因外键失效，后续 `GET /api/projects` 会返回孤儿项目——本版本不自动清理，依赖后端策略：<br>• 若后端在 DELETE 时**级联删除项目** → 前端刷新 store：`useProjectStore.byCompany` 对应公司项清空；<br>• 若后端**保留孤儿项目** → 不自动清理，下次拉取时剔除孤儿后展示（见 §2.6 项目列表过滤规则）；<br>`ElMessage.success('删除成功')` |
| 失败逻辑 | 409 → `ElMessageBox.alert(detail, '无法删除', { type: 'warning' })`；行保留 |

> 后端 DELETE 行为以 `backend/spec/task_management.md` 为准；前端按文档实测结果处理。本规格默认按「保留孤儿项目」落地。
>
> 删除公司成功后必须强制刷新**任务列表筛选条**的公司下拉：组件 unmount 时向外 emit `companies-changed`，由 `TaskList.vue` 调用 `useCompanyStore.fetchAll()` 兜底重拉字典（兜底即可，主流程以 store 内 reactive 更新为主）。

---

### 2.5 功能点：新建项目

#### 静态展示规则
- 字段：

| 字段 | 控件 | 必填 | 长度 | 默认值 |
|------|------|------|------|--------|
| name | `<el-input>` | ✓ | 1-128 | 空 |
| company_id | `<el-select>` | ✓ | — | 沿用顶部公司筛选值；若筛选项为空则为空 |
| notes | `<el-input>` | — | ≤ 4000 | 空 |

- 若顶部公司筛选有值，公司下拉默认值 = 顶部筛选值，且禁用（不可改）；
- 若顶部公司筛选项为空，公司下拉必选，否则按钮 disabled。

#### 交互逻辑

| 阶段 | 内容 |
|------|------|
| 操作前 | name / company_id / notes 已填 |
| 触发动作 | 点击「+ 新建」 |
| 接口请求 | `POST /api/projects` body：`{ name, company_id, notes? }` |
| 成功逻辑 | 项目列表追加新行；`useProjectStore.byCompany[company_id]` 同步追加；输入框清空（company 下拉恢复为顶部筛选值或空）；`ElMessage.success('创建成功')` |
| 失败逻辑 | 422 → 字段红字；<br>409 → name 红字「该项目名称已存在」；<br>5xx → toast |

---

### 2.6 功能点：编辑 / 删除项目

#### 静态展示规则与交互逻辑

与 §2.3 / §2.4 公司版完全一致，仅路径与字段差异：

- 编辑接口：`PUT /api/projects/{id}` body：`{ name, company_id, notes? }`；
- 删除接口：`DELETE /api/projects/{id}`；
- 删除文案：`确定删除「{name}」？若已被任务引用将无法删除。`；
- 删除 409 行为：被任务引用 → `ElMessageBox.alert` 展示后端 detail；列表行保留；
- 删除成功后：`useProjectStore.byCompany[old_company_id]` 中移除该项目；若项目所属公司发生变化（编辑改 company_id），需同步两个公司分组。

#### 列表过滤规则（孤儿项目）

- 后端在某些场景（如直接 DB 操作）可能返回 `company_id` 不存在的孤儿项目；
- 前端展示前做防御性过滤：`project.company_id` 不在当前 `useCompanyStore.companies` 中的项目**不展示**；
- 此类孤儿不弹提示（属于后端脏数据，不归用户感知）。

---

### 2.7 功能点：与字典的下游同步

任何写操作成功后，必须保证**任务表单下拉**与**任务列表筛选条**立即可见新数据。本模块通过 store 的 reactive 状态实现：

| 写操作 | store 同步 |
|--------|----------|
| 新建 / 编辑 / 删除公司 | `useCompanyStore.companies` 增删改 |
| 新建项目 | `useProjectStore.byCompany[company_id].push(item)` |
| 编辑项目（含改 company_id） | 旧分组移除，新分组 push；当前 store 不存在的新公司 → 调 `fetchByCompany` |
| 删除项目 | 旧分组 splice |

> 编辑项目改 `company_id` 是低频操作；store 不预加载所有公司项目（按公司懒加载是已有约定）。当用户改到一个未缓存的 `company_id` 时，触发 `projectStore.fetchByCompany(newCompanyId)`，获取后合并进 `byCompany[newCompanyId]`。

## 3. 测试案例（Playwright + Chrome）

### 3.1 用例编号约定

`TC<模块编号><序号>`：`TC-CP-01` … `TC-CP-NN`

### 3.2 用例清单

#### TC-CP-01 打开管理对话框
- **前置**：后端存在至少 1 家公司；
- **步骤**：
  1. 访问主页；
  2. 点击操作条 `管理公司项目`；
  3. 断言网络并行收到 `GET /api/companies` 与 `GET /api/projects`；
  4. 断言对话框标题「公司 / 项目管理」可见；
  5. 断言默认激活 Tab 为「公司」；
- **预期**：对话框打开，两 Tab 数据就绪。

#### TC-CP-02 新建公司（正常）
- **前置**：后端无同名公司；
- **步骤**：
  1. 打开对话框；
  2. 在公司 Tab 顶部 name 输入「TC 公司 1」，notes 输入「测试备注」；
  3. 点击「+ 新建」；
  4. 断言网络收到 `POST /api/companies` body `{ name: "TC 公司 1", notes: "测试备注" }`；
  5. 断言公司列表追加「TC 公司 1」一行；
  6. 断言输入框已清空；
  7. 断言 toast「创建成功」；
  8. 关闭对话框，打开 `+ 新建任务`抽屉，公司下拉包含「TC 公司 1」；
- **预期**：新增公司立即在任务表单下拉中可见。

#### TC-CP-03 新建公司（重名 409）
- **前置**：后端已存在「TC 公司 1」；
- **步骤**：
  1. 打开对话框；
  2. name 输入「TC 公司 1」；
  3. 点击「+ 新建」；
  4. 断言网络收到 `POST /api/companies` 响应 409；
  5. 断言 name 字段下方红字「该公司名称已存在」；
  6. 断言对话框未关闭；
- **预期**：阻断重名提交，给出明确错误。

#### TC-CP-04 编辑公司
- **前置**：公司列表中存在公司 X；
- **步骤**：
  1. 在 X 行点击 `编辑`；
  2. 断言该行变为编辑态（出现 `保存` / `取消` 按钮）；
  3. 修改 name 为「TC 公司 1-改」；
  4. 点击 `保存`；
  5. 断言网络收到 `PUT /api/companies/{X}` body 含 `name="TC 公司 1-改"`；
  6. 断言列表 X 行 name 更新为「TC 公司 1-改」；
  7. 断言 toast「更新成功」；
- **预期**：编辑生效，下游字典同步。

#### TC-CP-05 删除公司（被引用 409）
- **前置**：公司 X 下存在项目 P，且 P 被某任务引用；
- **步骤**：
  1. 打开对话框；
  2. 在 X 行点击 `删除`；
  3. 断言 `ElMessageBox.confirm` 出现，文案含 X.name；
  4. 点击「确定」；
  5. 断言网络收到 `DELETE /api/companies/{X}` 响应 409；
  6. 断言 `ElMessageBox.alert` 出现，type='warning'，文案含后端 detail；
  7. 断言 X 仍在列表中；
- **预期**：删除被阻断，提示明确。

#### TC-CP-06 删除公司（确认）
- **前置**：公司 X 无任何项目 / 任务引用；
- **步骤**：
  1. 在 X 行点击 `删除`；
  2. 在确认弹窗中点击「确定」；
  3. 断言网络收到 `DELETE /api/companies/{X}`；
  4. 断言 X 行从列表移除；
  5. 断言 toast「删除成功」；
  6. 关闭对话框，打开任务表单，公司下拉已不含 X；
- **预期**：删除生效，下游字典同步。

#### TC-CP-07 新建项目
- **前置**：公司 A 存在；
- **步骤**：
  1. 切换到「项目」Tab；
  2. 顶部公司筛选下拉选择 A；
  3. name 输入「TC 项目 1」，notes 输入「备注」；
  4. 点击「+ 新建」；
  5. 断言网络收到 `POST /api/projects` body 含 `company_id=A`、`name="TC 项目 1"`；
  6. 断言项目列表追加「TC 项目 1」一行；
  7. 断言 toast「创建成功」；
  8. 关闭对话框，打开任务表单，公司 A 下项目下拉含「TC 项目 1」；
- **预期**：新增项目立即在项目下拉中可见。

#### TC-CP-08 新建项目（同公司重名 409）
- **前置**：公司 A 下已存在「TC 项目 1」；
- **步骤**：
  1. 切到项目 Tab，顶部筛选选 A；
  2. name 输入「TC 项目 1」；
  3. 点击「+ 新建」；
  4. 断言响应 409；
  5. 断言 name 红字；
- **预期**：同公司下项目名唯一，跨公司允许重名。

#### TC-CP-09 编辑项目（含跨公司）
- **前置**：公司 A 下有项目 P1，公司 B 下无 P1；
- **步骤**：
  1. 切到项目 Tab，顶部筛 A；
  2. P1 行点 `编辑`；
  3. 将 company 改为 B，name 保持不变；
  4. 点击 `保存`；
  5. 断言响应 200；
  6. 顶部筛 A，列表不应再有 P1；
  7. 顶部筛 B，列表应出现 P1；
  8. 断言 B 下项目下拉（在任务表单中）含 P1；
- **预期**：跨公司迁移生效，字典同步。

#### TC-CP-10 删除项目（被任务引用 409）
- **前置**：项目 P 被至少一个任务引用；
- **步骤**：
  1. 切到项目 Tab；
  2. 在 P 行点击 `删除`；
  3. 确认弹窗点击「确定」；
  4. 断言响应 409；
  5. 断言 `ElMessageBox.alert` 出现，文案含后端 detail；
  6. 断言 P 仍在列表中；
- **预期**：删除被阻断。

#### TC-CP-11 项目列表按公司筛选
- **前置**：A 下有 P1、B 下有 P2；
- **步骤**：
  1. 切到项目 Tab；
  2. 默认筛选为空，列表显示 P1 + P2；
  3. 顶部筛 A，列表仅显示 P1；
  4. 顶部筛 B，列表仅显示 P2；
  5. 清空筛选，列表恢复 P1 + P2；
- **预期**：筛选生效。

#### TC-CP-12 关闭对话框不请求任务列表
- **功能点**：弹窗关闭不影响主页数据
- **步骤**：
  1. 打开对话框；
  2. 关闭对话框（点 X / ESC / 遮罩）；
  3. 断言主页未发出额外的 `GET /api/tasks` 请求；
- **预期**：弹窗与主页独立。

#### TC-CP-13 字典加载失败
- **前置**：通过 `page.route` 拦截 `GET /api/companies` 返回 500；
- **步骤**：
  1. 点击 `管理公司项目`；
  2. 断言对话框不打开；
  3. 断言出现 toast（5xx 文案）；
- **预期**：加载失败阻断弹窗打开，给出错误提示。

### 3.4 测试数据管理

- **隔离**：每个 `test` 用唯一后缀（如 `Date.now()`）避免冲突；
- **清理**：在 `globalTeardown` 中调用后端 `DELETE` 或重置测试库；
- **夹具（fixture）**：常用字典（公司 / 项目）放在 `e2e/fixtures/seed.js`，通过 `test.beforeAll` 调用后端 API 灌入；
- **不依赖顺序**：每个用例独立前置条件，不假定前置 case 已跑过。
